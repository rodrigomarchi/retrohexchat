# Implementation Plan: P2P Lobby & Session UI

**Branch**: `035-p2p-session-ui` | **Date**: 2026-02-16 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/035-p2p-session-ui/spec.md`

## Summary

Build the user-facing layer for P2P sessions: slash commands (`/p2p`, `/call`, `/sendfile`) to initiate sessions with dual-channel invitations (PM + toast), a dedicated lobby LiveView (`/p2p/:token`) with ephemeral chat, peer presence indicators, and bilateral consent for actions (file transfer, audio/video calls). Extends the existing P2P bounded context (feature 034) with lobby state in the SessionServer GenServer and adds browser capability detection via JS hooks. Post-consent transitions to "connecting" state as a handoff point for future WebRTC implementation.

## Technical Context

**Language/Version**: Elixir 1.17+ / OTP 27+, JavaScript ES2020+
**Primary Dependencies**: Phoenix 1.8+, Phoenix LiveView 1.0+, 98.css (npm), esbuild
**Storage**: PostgreSQL 16+ (existing `p2p_sessions` table, existing `private_messages` table — no new migrations). GenServer state for ephemeral lobby data.
**Testing**: ExUnit (unit/integration/liveview), Vitest + jsdom (JS hooks/lib)
**Target Platform**: Web (modern browsers with WebRTC support)
**Project Type**: Umbrella web application (existing)
**Performance Goals**: PM delivery <1s, lobby chat <500ms, capability detection <2s (non-blocking)
**Constraints**: Ephemeral lobby chat only (no DB persistence), 100 message cap in GenServer, 60s action request timeout
**Scale/Scope**: 2 concurrent peers per session, multiple sessions per user (with different peers)
